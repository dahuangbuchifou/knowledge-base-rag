## 🎯 第一阶段任务拆解

### 任务 1：搜索优化（关键词 + 语义）

#### 1.1 当前问题
- 用户只能提问，无法主动搜索
- 无搜索框、无搜索建议
- 无搜索结果展示

#### 1.2 目标
- 添加搜索框，支持关键词搜索
- 支持语义搜索（理解用户意图）
- 搜索结果展示（摘要 + 来源 + 时间）

#### 1.3 技术方案

**方案 A：前端搜索（简单，适合小数据量）**
```
用户输入 → 前端搜索本地缓存 → 显示结果
```
- **优点：** 实现简单，响应快
- **缺点：** 只能搜索已缓存的内容
- **适合：** 知识库内容少，用户问题重复率高

**方案 B：后端搜索（推荐）**
```
用户输入 → 后端百炼 API → 返回搜索结果
```
- **优点：** 搜索完整知识库，准确度高
- **缺点：** 需要 API 调用，响应稍慢
- **适合：** 知识库内容丰富，需要准确搜索

**方案 C：混合搜索（最优）**
```
用户输入 → 前端缓存 + 后端 API → 合并结果
```
- **优点：** 兼顾速度和准确度
- **缺点：** 实现复杂
- **适合：** 长期方案

#### 1.4 实施步骤（方案 B - 推荐）

**Step 1：后端添加搜索接口**
```javascript

    const { query, filters } = req.body;
    
    // 调用百炼 API 搜索
    // 返回：{ results: [{ title, summary, source, date, score }] }
});
```

**Step 2：前端添加搜索框**
```html
<!-- index.html 新增 -->
<div class="search-box">
    <input type="text" id="searchInput" placeholder="搜索知识库...">
    <button onclick="search()">搜索</button>

```

**Step 3：搜索结果展示**

**Step 4：搜索建议（可选）**

#### 1.5 注意事项
- 百炼 API 是否支持搜索功能？需确认
- 如果不支持，需要自建向量数据库（ChromaDB/Milvus）
- 搜索结果排序算法（相关度 + 时间）

#### 1.6 预期效果
- 用户能主动搜索问题

### 任务 2：答案引用来源

#### 2.1 当前问题
- 回答无来源标注
- 用户不知道答案从哪来
- 无法核对原文

#### 2.2 目标
- 每个答案附带来源链接
- 区分"来自知识库"和"模型推测"
- 点击可展开原文片段

#### 2.3 技术方案

**Step 1：后端开启引用**
```javascript
// server.js 修改
const postData = JSON.stringify({
    input: { prompt: question },
    parameters: {
        has_thoughts: true  // 开启引用
    }
});
```

**Step 2：解析引用数据**
```javascript
// 百炼 API 返回格式（假设）
{
    output: {
        text: "答案内容...",
        references: [
            {
                title: "GB17691-2018",
                page: 15,
                snippet: "原文片段...",
                confidence: 0.95
            }
        ]
    }
}
```

**Step 3：前端展示引用**
```html
<div class="answer">
    <div class="content">答案内容...</div>
    <div class="references">
        <div class="reference">
            📄 GB17691-2018 | 第 15 页
            <button onclick="showSnippet()">展开原文</button>
        </div>
    </div>
</div>
```

#### 2.4 注意事项
- 需确认百炼 API 返回的引用格式
- 如果 API 不支持引用，需要自建 RAG 系统
- 引用置信度阈值（< 0.8 标记为"推测"）

#### 2.5 预期效果
- 每个答案附带来源
- 低置信度答案有提示
- 答案引用覆盖率 > 90%

### 任务 3：内容分类和模板

#### 3.1 当前问题
- 知识库内容无分类
- 文档格式不统一
- 无负责人、无更新时间

#### 3.2 目标
- 建立知识分类体系
- 统一文档模板
- 每篇文档有分类、标签、负责人、更新时间

#### 3.3 技术方案

**Step 1：分类体系设计**
```
发动机排放标准知识库
├── 国家标准
│   ├── GB17691-2018（重型车）
│   └── GB/T17692-2024（测试方法）
├── 测试标准
│   ├── WHSC 测试
│   ├── WHTC 测试
│   └── WHTC+WHSC 测试
├── 限值标准
│   ├── NOx 限值
│   ├── PM 限值
│   └── CO 限值
└── 常见问题
    ├── 测试流程
    ├── 公式计算
    └── 标准解读
```

**Step 2：文档模板**
```markdown
# [文档标题]

## 基本信息
- **分类：** [一级分类] / [二级分类]
- **标签：** [标签 1, 标签 2, 标签 3]
- **负责人：** [姓名]
- **更新时间：** [YYYY-MM-DD]
- **版本：** [v1.0]

## 内容
[文档内容]

## 参考资料
- [参考 1]
- [参考 2]
```

**Step 3：知识库更新**
- 现有文档按分类整理
- 补充标签、负责人、更新时间
- 上传到百炼知识库

#### 3.4 注意事项
- 分类不宜过细，3-5 个一级分类即可
- 标签要统一，避免同义词
- 负责人要明确，方便后续更新

#### 3.5 预期效果
- 知识分类清晰
- 文档格式统一
- 方便后续扩展和权限管理

### 任务 4：用户反馈机制

#### 4.1 当前问题
- 无用户反馈渠道
- 不知道哪些问题没答好
- 无法收集用户意见

#### 4.2 目标
- 每个答案支持"有用/无用"
- 收集用户未解决问题
- 后台形成待优化任务

#### 4.3 技术方案

**Step 1：前端反馈按钮**
```html
<div class="answer-footer">
    <button class="thumbs-up" onclick="feedback('useful')">👍 有用</button>
    <button class="thumbs-down" onclick="feedback('useless')">👎 无用</button>
    <button class="suggest" onclick="showSuggestForm()">💡 补充建议</button>
</div>
```

**Step 2：后端反馈接口**
```javascript
// server.js 新增
app.post('/api/feedback', (req, res) => {
    const { question, answer, rating, comment } = req.body;
    
    // 保存到日志/数据库
    fs.appendFileSync('feedback.log', JSON.stringify({
        time: new Date().toISOString(),
        question,
        answer,
        rating,  // 'useful' | 'useless'
        comment
    }) + '\n');
    
    res.json({ success: true });
});
```

**Step 3：后台统计（可选）**
```javascript
// 定时任务：分析反馈数据
// 生成：热门问题、无结果问题、差评答案
```

#### 4.4 注意事项
- 反馈数据要匿名（不收集用户身份）
- 差评答案要标记，方便后续优化
- 定期分析反馈数据，形成优化任务

#### 4.5 预期效果
- 用户能反馈答案质量
- 收集到真实问题
- 反馈提交量 > 10 次/周
